実装の精査からコミット・プッシュまでを一括実行してください。

## 実行手順

### ステップ 1: コード精査・修正（自動適用）

変更されたファイルを対象に、以下の観点で精査し、問題があれば**ユーザー確認なしで即修正**を適用する。

- 重複コード・共通化できる処理の抽出
- 不要な変数・インポート・コメントの削除
- 明らかなバグ・型の不整合
- パフォーマンス上の問題（不要な再レンダリング・重い処理など）
- セキュリティ上の問題（XSS・インジェクションなど）

問題がなければスキップして次のステップへ進む。

### ステップ 2: ドキュメント更新

.agent/workflows/update-docs.md の手順に従い、以下を更新する。

1. `CHANGELOG.md` の `[Unreleased]` セクションに変更内容を追記（ユーザー視点・実装詳細は書かない）
2. `ROADMAP.md` を更新（完了タスクを「進行中」から削除して CHANGELOG へ移動、項目数上限を遵守）
3. `.agent/SESSION.md` を更新（Last Updated・Current Focus・Recent Achievements）

### ステップ 3: コミット

以下をすべて自動で実行する（ユーザー確認は不要）。

1. `git status` と `git diff` で変更内容を確認する
2. AGENTS.md のコミット規約に従いコミットメッセージを自動生成する
   ```
   [種別] 変更内容の要約（日本語・動詞で締める）

   - 変更点1
   - 変更点2
   ```
3. `git add .` や `git add -A` は使わず、対象ファイルを個別に `git add` する
4. `git commit` を実行する
5. `git push origin main` でプッシュする

### ステップ 4: Issue の自動クローズ

以下を実行して、今回の変更で解決された Issue を自動判断してクローズする。

1. `gh issue list --limit 50 --state open` で Open Issues をすべて取得する
2. 各 Issue のタイトル・本文と今回の変更内容（git diff の概要）を照合する
3. 「この変更によって Issue の問題が解決された」と判断できる Issue を特定する
   - 判断基準: Issue のタイトル・本文に記載された症状・機能要求が、今回の変更で対応されているか
   - 確信が持てない場合はクローズしない（過剰クローズを避ける）
4. 該当 Issue があれば以下を実行する：
   ```bash
   gh issue close <番号> --comment "修正内容の一言説明"
   ```

## 完了報告

以下を日本語で報告する。

- 精査で修正したファイルと内容（修正なしの場合は「精査：問題なし」）
- 更新したドキュメント一覧
- コミットハッシュとコミットメッセージ
- Close した Issue 番号（該当する場合）

# GitHub Issues 運用

## この文書の役割
- 不具合、要望、気づき、内部タスクを GitHub Issues に集約する
- どのテンプレートを使うか、どう整理するかを簡単にそろえる

## 基本方針
- 未解決の課題は GitHub Issues を正本として扱う
- `docs/dev/operations/不具合・要望リスト.md` は今後更新しない
- `ROADMAP.md` は高レベルな進行中 / 予定だけを書く
- 完了した変更は `CHANGELOG.md` に残す

## 使うテンプレート
- `🐛 バグ報告`
  - 再現する不具合、動かない挙動、保存されない状態など
- `✨ 要望`
  - 新しい機能、操作導線の追加、使い勝手の改善提案
- `💡 気づき・改善案`
  - まだ仕様化していない違和感、観察メモ、改善のヒント
- `🛠 改善・内部タスク`
  - 実装、調査、リファクタ、ドキュメント更新、リリース準備

## 推奨ラベル
- 種別
  - `バグ`
  - `要望`
  - `気づき`
  - `改善`
- 優先度
  - `優先度:高`
  - `優先度:中`
  - `優先度:低`
- 状態
  - `要トリアージ`
  - `調査中`
  - `対応中`
  - `確認待ち`

## 運用の流れ
1. まず Issue を作る
2. 種別ラベルと優先度ラベルを付ける
3. 必要なら GitHub Projects で `未着手 / 調査中 / 対応中 / 完了` に並べる
4. 実装時は Issue 番号をブランチ名、コミット、PR に紐づける
5. 完了後は Issue を閉じ、必要なら `CHANGELOG.md` に反映する

## 判断の目安
- すぐ直す / 直したいもの
  - GitHub Issue にする
- まだ曖昧だが残したい気づき
  - GitHub Issue の `気づき・改善案` を使う
- 今回は見送るが将来候補として残したい
  - GitHub Issue にしたうえで `ROADMAP.md` の将来候補へ要約だけを書く

## GitHub Projects のおすすめ列
- 未着手
- 調査中
- 対応中
- 確認待ち
- 完了

## 既存ドキュメントとの関係
- `README.md`
  - GitHub Issues への案内を載せる
- `AGENTS.md`
  - セッション終了時に GitHub Issues 更新要否を確認する
- `ROADMAP.md`
  - 詳細管理ではなく、直近の重点項目だけを書く
- `CHANGELOG.md`
  - 解決済みの変更結果だけを書く

# 自動更新検討メモ v1

作成日: 2026-03-06

## 目的
- 現行の手動更新（`update.bat`）運用を棚卸しし、自動更新導入の現実的な進め方を決める。

## 現状（実装ベース）
- 配布形式:
  - `electron-builder` の `win.target` は `zip` のみ。
- 更新導線:
  - ユーザーが `update.bat` を実行し、ZIPを選択して更新。
  - `update.bat` が `MediaArchiver v2.exe` を終了し、展開済みファイルを上書きして再起動。
- アプリ本体:
  - `autoUpdater` / `electron-updater` の導入コードは未実装。
  - `electron-squirrel-startup` は無効化済み（ディレクトリ配布前提）。

## 現行方式の利点と課題

### 利点
- 既存ユーザー運用に合っている（ZIP差し替え）。
- 署名・配信基盤が未整備でも更新できる。
- 実装の複雑性が低い。

### 課題
- 更新は手動操作が前提で、更新通知・自動取得がない。
- ZIP選択ミスや実行場所ミスの運用リスクが残る。
- 差分更新ではなく全量更新になりやすい。

## 選択肢

### 案A: ZIP運用を維持しつつ「更新検知 + ダウンロード」だけ自動化
- 内容:
  - アプリ内で最新版を確認（manifest取得）
  - ダウンロードまではアプリ側で実施
  - 適用は `update.bat` へ受け渡し（再起動を含む）
- 長所:
  - 既存配布形式を維持できる
  - 移行コストが最小
- 短所:
  - 最終適用はバッチ依存のまま
  - 完全自動更新ではない

### 案B: インストーラー配布（例: NSIS）+ 自動更新基盤へ移行
- 内容:
  - 配布形式をインストーラー中心へ変更
  - 更新適用もアプリ内導線へ統合
- 長所:
  - 更新体験を一本化しやすい
  - 将来的に差分更新運用へ寄せやすい
- 短所:
  - 移行コスト・検証コストが高い
  - 署名・配信運用の整備が必要

## 推奨方針（v1）
- 段階導入で案Aから開始し、運用が安定したら案Bを再評価する。

## 実装ステップ案
1. 棚卸し完了（この文書）
2. 更新フィード仕様を決める
   - 例: `version / releaseNotes / downloadUrl / publishedAt / sha256`
3. アプリ内「更新確認」PoC
   - 手動ボタンで最新版チェック
   - 結果をダイアログ表示
4. ダウンロードPoC
   - ZIPを一時領域へ保存
   - ハッシュ照合
5. 適用PoC
   - `update.bat <downloaded_zip_path>` 呼び出し
   - 再起動確認

## 実施済み（2026-03-06）
- ステップ3（更新確認PoC）を実装した。
  - Electron main: GitHub Releases 最新版を参照する更新確認サービスを追加。
  - 設定画面（一般タブ）: `更新を確認（PoC）` ボタンを追加。
  - 結果表示: 現在バージョン / 最新バージョン / 更新あり・なし / 失敗理由 / リリースノート本文。
- ステップ4（ダウンロードPoC）を実装した。
  - 設定画面（一般タブ）に `更新ZIPを取得（PoC）` を追加。
  - 最新リリースの ZIP 資産を一時フォルダへ保存し、SHA-256 を計算して保持。
  - `.sha256` 資産がある場合は照合し、不一致時は保存ファイルを破棄。
- ステップ5（適用PoC）を実装した。
  - 設定画面（一般タブ）に `update.bat で適用（PoC）` を追加。
  - ダウンロード済み ZIP のパスを引き渡して `update.bat <zip_path>` を起動できるようにした。

## PoC完了条件（最小）
- 開発環境で「更新あり」判定が再現できる。
- ダウンロードしたZIPを `update.bat` に渡して更新完了まで到達できる。
- 失敗時にユーザーへ復旧手順（再試行・手動更新）を提示できる。

## 主なリスク
- 配信元の可用性・改ざん耐性（ハッシュ検証前提）。
- 更新中のプロセスロック（AV干渉含む）。
- 権限不足での上書き失敗。

## 検証観点
- 正常系:
  - 更新なし
  - 更新あり→ダウンロード→適用→再起動
- 異常系:
  - manifest取得失敗
  - ZIP破損
  - ハッシュ不一致
  - 適用失敗（コピー失敗/ロック）

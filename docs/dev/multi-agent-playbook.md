# Multi-Agent Playbook

## 目的
- このリポジトリでマルチエージェント運用をするときの役割分担を固定し、調査・実装・検証の衝突を減らす
- `AGENTS.md` の運用ルールを前提に、複数の作業セッションでも判断基準がずれにくい状態を作る

## 前提
- 役割の本体は「設定ファイル」ではなく「役割ごとの専用プロンプト」
- 全役割が `AGENTS.md` を読む前提で動く
- 同じファイルを複数役が同時に編集しない
- 最終的な統合、ドキュメント更新、コミット、プッシュは親エージェントか人間が担当する

## どんな作業で使うか

### 向いている作業
- Electron main / preload / renderer をまたぐ変更
- Lightbox / Center Viewer / RightPanel のように関連箇所が広い改修
- 仕様確認と回帰確認が重い UI 改修
- 実装と検証、ドキュメント更新を分離したい作業

### 向いていない作業
- 1ファイルで閉じる軽微な修正
- 文言修正や小さいスタイル調整
- 同じ関数群を細かく触る密結合リファクタ

## 役割

### 親エージェント
- 目的:
  - タスクを調査・実装・検証に分割し、完了条件を固定する
- やること:
  - 依頼単位を切る
  - 担当ごとの触る範囲を決める
  - 競合が出た場合に優先順位を決める
  - 最終差分を統合し、`ROADMAP.md` `CHANGELOG.md` `docs/dev/不具合・要望リスト.md` の更新要否を判断する
- やらないこと:
  - 仕様が固まる前に複数役へ同時実装させる
  - 同じファイルを複数役へ編集依頼する

### 調査役
- 目的:
  - 影響範囲、既存仕様、変更時の注意点を整理する
- やること:
  - 関連ファイルを列挙する
  - 実装上の根拠を抜き出す
  - 既存ドキュメントと現コードの差を洗う
  - 未確認点を明示する
- やらないこと:
  - コード編集
  - 推測だけでの仕様断定

### 実装役
- 目的:
  - 完了条件に沿って最小変更でコードを直す
- やること:
  - 指定された範囲を実装する
  - 必要な確認コマンドを実行する
  - 未解決点を残す場合は明示する
- やらないこと:
  - 指定外の広いリファクタ
  - 目的外の UI 変更や命名変更

### 検証役
- 目的:
  - 回帰、仕様抜け、確認漏れを見つける
- やること:
  - 変更後の動作を review 観点で点検する
  - 再現手順を添えて findings を出す
  - テスト不足や未確認領域を明示する
  - ドキュメント更新漏れを指摘する
- やらないこと:
  - 要約だけで終わる報告
  - 根拠のない「問題なし」判定

## このリポジトリで参照する基準文書
- `AGENTS.md`
  - 文字コード、PowerShell、ドキュメント更新、コミット運用
- `docs/dev/lightbox-redesign-v2.md`
  - Lightbox / Center Viewer / RightPanel の責務基準
- `docs/dev/回帰確認チェックリスト.md`
  - UI 変更時の最低限の確認観点
- `docs/dev/不具合・要望リスト.md`
  - 未解決課題と残件管理
- `ROADMAP.md`
  - 進行中と予定の管理
- `CHANGELOG.md`
  - 完了した変更の記録

## 基本フロー
1. 親エージェントがテーマを1つに絞る
2. 調査役が影響範囲と注意点を確定する
3. 親エージェントが完了条件と編集範囲を固定する
4. 実装役がコード変更と最低限の確認を行う
5. 検証役が回帰と残課題を洗う
6. 親エージェントが必要なドキュメント更新を反映し、コミットとプッシュを行う

## 依頼テンプレート

### 親エージェント -> 調査役
```text
目的:
- [今回のテーマ]

調べること:
- 既存仕様
- 関連ファイル
- 変更時の注意点

制約:
- コードは編集しない
- 根拠がある内容だけを書く

出力形式:
- 関連ファイル
- 現状
- 注意点
- 未確認点
```

### 親エージェント -> 実装役
```text
目的:
- [今回のテーマ]

完了条件:
- [満たすべき状態]

編集範囲:
- [触ってよいファイルや層]

制約:
- 指定外のリファクタはしない
- AGENTS.md の運用ルールに従う

出力形式:
- 変更内容
- 変更ファイル
- 確認結果
- 残課題
```

### 親エージェント -> 検証役
```text
目的:
- [今回のテーマ]

確認対象:
- [変更点]

制約:
- findings を優先する
- UI 変更では docs/dev/回帰確認チェックリスト.md を基準にする

出力形式:
- Findings
- 再現手順
- 未確認点
- ドキュメント更新要否
```

## 受け渡しフォーマット

### 調査役の出力
- 関連ファイル
- 現仕様
- 変更時の注意点
- 未確認点

### 実装役の出力
- 変更内容
- 変更ファイル
- 実行した確認
- 残課題

### 検証役の出力
- Findings
- 再現手順
- リスク
- 未確認点
- ドキュメント更新要否

## 競合を防ぐルール
- 同じファイルを複数役へ同時に割り当てない
- 仕様が未確定の段階では実装役を動かさない
- 調査結果なしで `main / preload / renderer` 横断改修を始めない
- UI 変更時は、検証役に画面幅とデータ種別の確認観点を必ず渡す
- 途中で仕様が変わったら、親エージェントが依頼を出し直す

## このプロジェクトでのおすすめ分け方

### Lightbox / Center Viewer 改修
- 調査役:
  - `docs/dev/lightbox-redesign-v2.md` と現実装差分の確認
- 実装役:
  - `src/features/center-viewer/` と `src/components/RightPanel/` の修正
- 検証役:
  - 閉じる、前後移動、右パネル重複表示、画像/動画/書庫の確認

### パフォーマンス改善
- 調査役:
  - 遅延読込境界、依存の重い箇所、既存計測の把握
- 実装役:
  - chunk 分割や遅延読込の修正
- 検証役:
  - 起動直後、一覧初回表示、右パネル初回表示、Center Viewer 初回表示の体感差確認

### 一覧 UI 調整
- 調査役:
  - どの表示モードで崩れるかの洗い出し
- 実装役:
  - `FileGrid` `FileCard` `Header` 周辺の調整
- 検証役:
  - `docs/dev/回帰確認チェックリスト.md` のスモーク確認

## 最低限の運用ルール
- 小さい修正は単独で終わらせる
- マルチエージェント化するのは中規模以上の変更だけに絞る
- 実装完了後は必ずドキュメント更新とコミット運用まで含める
- 判断が割れたら、親エージェントが「変更を減らす方向」で決める

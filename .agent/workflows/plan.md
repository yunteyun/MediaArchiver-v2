# 実装計画立案ワークフロー

機能の設計・実装計画を立案する際は、以下の手順に従ってください。

## 1. 要件の整理

- ユーザーの要求を明確化
- 関連する既存コードの調査
- 影響範囲の特定

## 2. 設計判断（必要に応じて）

重要な設計判断がある場合は、ADR を作成：
- `.agent/decisions/XXX-判断名.md`

## 3. 実装計画の提示

**テンプレートを使用してください**: `.agent/templates/implementation_plan_template.md`

以下を含む計画をユーザーに提示する：

### 必須項目
- **概要**: 対象機能、目的
- **実装スコープ**: やること / やらないことの明確化
- **UI仕様**: 対象画面、操作フロー、UI制約
- **状態管理**: 使用する state、禁止事項
- **非同期処理 / IPC / FS**: 使用有無と制約
- **影響範囲**: 変更されるファイル、変更されないもの
- **リスクと対策**: 想定リスク、回避策
- **完了条件**: 検証方法

> **AGENTS.md に準拠しない計画は実装しない。**
> 「禁止事項」「UI制約」に抵触しないか必ず確認してください。

## 4. ユーザーレビューを受けてから実装を開始

計画提示後はユーザーの承認を待つ。承認なしに実装を開始しない。

## 5. SESSION.md の更新

計画立案後、`.agent/SESSION.md` の Next Steps を更新する。

---

## 精査の要否判断

### 精査あり（推奨）
- DB 設計、コア機能、複雑なロジック
- 複数ファイルにまたがる大きな変更

### 精査なし（軽微な変更時）
- UI の微調整
- ドキュメント更新
- 明らかなバグ修正
- 影響範囲が限定的な小タスク
